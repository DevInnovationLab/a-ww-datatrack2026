# CLAUDE.md

Guidance for Claude Code when working in this repository.

## What this repo is

A teaching pipeline (DIL Welcome Week session) that ingests a messy raw household water/chlorine survey export and turns it into tidy, labeled, de-identified Stata datasets. See [readme.md](readme.md) for the full pipeline description, per-script inputs/outputs, and a list of known inconsistencies between `main.do` and the actual scripts in `code/`.

## Keeping the README in sync

`readme.md` documents, per script in `code/`: its inputs, its outputs (including the `.md` reports and `.xlsx` codebooks generated by `iesave`/`iecodebook`), and a one-paragraph summary of what it does. Whenever you add, rename, delete, or meaningfully change the behavior of a script in `code/`, update the matching section of `readme.md` in the same change — do not leave it to drift. Specifically:

- **New or renamed `.do` file in `code/`:** add/update its entry under "Description of programs/code", and update the pipeline diagram if it changes the data flow.
- **Change to a script's inputs, outputs, or the transformations it performs:** update that script's bullet points (inputs/task/outputs) to match. Do not include .md files from iesave in the outputs.
- **New or renamed data file under `data/`:** update the relevant table under "List of datasets" (raw / tidy / clean).
- **Fixing `main.do` or `04-codebook-verify.do` to match the current file/variable names:** remove the corresponding bullet from "Known inconsistencies" once it's actually fixed — don't leave stale warnings after the underlying problem is resolved.
- **Discovering a new inconsistency** (a script referencing a file/variable that doesn't exist, dead code, missing outputs): add it to "Known inconsistencies" rather than silently working around it, so the next person (human or Claude) doesn't have to re-derive it.

## Project conventions

- **The one rule:** cleaning/tidying scripts here change data *shape* and *format* only — never *values*. Duplicates, outliers, and contradictions in the raw data are intentionally preserved for a later HFC session. Don't "fix" data issues in these scripts; that's out of scope.
- User-written Stata commands live in `code/ado/i` and are added to `adopath` by `main.do`; there's no package-install step to run.
- Follow the general folder-structure and PR/branch conventions in [CONTRIBUTING.md](CONTRIBUTING.md) (based on the DIL project template) unless the user says otherwise.

# Stata code conventions

- Prefer to use `iesave` over `save` in Stata, unless there's an explanation for why it is not done (e.g. unreachable/legacy code, noted with a comment).
- Code should be organized in numbered sections. Section headers should be indexed using `**# N Title`, with a `**------` divider line above and below; subsections use `**## N.M Title`. Keep numbering sequential and accurate to what's actually in the file — don't leave gaps or copy-pasted numbers/titles from another script.
- Indent with tabs, not spaces.
- Every script's header should list: author(s), last-updated date, inputs, outputs, a one-paragraph summary, and a "Notes" block for anything a reader wouldn't get from the code alone (e.g. why a sentinel code maps where it does, why a file is stale/dead code, where labels are sourced from).

## Labeling conventions

- Use `lab var` (not `label variable`) to label variables.
- Where a variable comes from a specific question in `documentation/Household Water Questionnaire - V1.xlsx` (the SurveyCTO form), put that question's code at the **front** of the label, e.g. `lab var hh_id "(A1) Household ID"` — not `"Household ID (A1)"`. Metadata variables that aren't survey questions (`key`, `child_index`, `submissiondate`, ...) get no code.
- Question codes are `<section letter><question number within that section>` (e.g. `C6` is the 6th question in Section C), derived from the form's `survey` sheet, not copied from any older/stale label text already in a script.
- Group `lab var` calls into subsections that mirror the questionnaire's lettered sections (e.g. `**## 6.4 Section C: Respondent & household roster`), plus one subsection for non-survey metadata.
- Keep every label at or under Stata's 80-character `label variable` limit — check with a quick script when adding a batch of labels.
- Note the labels' source (the CTO form path) in the script's header Notes.

## `main.do` conventions

- For each `do "${code}/....do"` call, keep the comment block immediately above it in sync with that script: a one-line summary of what it does, plus `Inputs:` and `Outputs:` matching the script's own header. Update these in the same change whenever a script's behavior, inputs, or outputs change.
- If a script is known-stale (see readme.md > Known inconsistencies), say so inline in its comment block (e.g. `04-codebook-verify.do [STALE -- see readme.md]`) rather than describing it as if it worked.
- Keep the header's `Outline` step list (script names and short descriptions) matching the actual `do` calls below it — don't let it drift back to old filenames or exercise-stub framing once a script is completed.
